kimi回答

不知道为什么将target修改为2就行了，问了一下kimi,kimi回答如下：
Target 2 (Native upload) 之所以成功，是因为它绕过了 Windows Server 2016 对 PowerShell 的严格限制。

各 Target 的技术差异

Target ID 名称 工作原理 依赖环境
0 Automatic 自动选择，通常优先尝试 PowerShell 取决于系统配置
1 PowerShell 生成 PowerShell 代码执行 payload PowerShell 执行策略、AMSI、Defender
2 Native upload 直接上传 EXE 文件，创建系统服务执行 仅需 SMB 和创建服务权限
3 MOF upload 通过 WMI/MOF 文件执行（旧技术） WMI 服务可用性

为什么 PowerShell (Target 0/1) 在 Server 2016 上失败

Windows Server 2016 默认启用了多层防护，导致 PowerShell payload 容易被拦截：

1. PowerShell 执行策略 (Execution Policy)

2. AMSI (Anti-Malware Scan Interface)

  • PowerShell 5.0+ 内置 AMSI，会实时扫描脚本内容
  • Metasploit 的 PowerShell payload 通常会被标记为恶意
  • 你的日志中出现 Service start timed out，说明 PowerShell 进程被 Defender 拦截或终止

3. Windows Defender 实时保护

  • 现代 Windows Server 默认启用 Defender
  • PowerShell 下载执行操作（DownloadString/Invoke-Expression）是常见检测特征

Native Upload (Target 2) 为什么能成功

原理完全不同，它不依赖 PowerShell 解释器：

优势：

  • ✅ 不经过 PowerShell：绕过脚本层面的所有检测机制
  • ✅ 原生 API：直接调用 Windows Service Control Manager，这是系统核心功能，无法禁用
  • ✅ 文件级操作：上传的是普通 PE 文件（.exe），不像 PowerShell 代码那样容易被 AMSI 静态扫描

渗透测试建议

针对 Windows Server 2012/2016/2019/2022：

  1. 优先尝试 Target 2 (Native upload)：成功率最高，兼容性最好
  2. 如果 Native 失败：尝试 Target 3 (MOF upload) 或改用 ms17_010_eternalblue 模块
  3. PowerShell 目标仅适用于：旧版系统（Win 7/2008）或已知关闭 Defender 的环境

总结：Target 2 成功是因为它把攻击从”脚本层”降到了”二进制/服务层”，绕过了 Server 2016 对 PowerShell 的现代防护机制。
